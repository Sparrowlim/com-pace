## Goal

쪼개기 완료 후 큐에 조각이 2개 이상이면 사용자가 "만만한 것"을 직접 탭해서 고르고(SPEC §3·D-05), 15분 종료 후 회고에서 완료/미완료 무관하게 "체감 어땠나요?" 3버튼(D-11)이 항상 뜨는 상태. 기존 PH-05/06/07 산출물(라우팅·PredictPage·FocusPage·세션 복구·온보딩 게이트)은 전부 그대로.

## 착수 전 설계 결정 (구현 전 확정 — 아래 체크리스트의 전제)

1. **선택 UI는 새 화면이 아니라 대시보드의 기존 "쪼개짐 + 큐 有" 분기를 확장한다.** `DashboardPage`가 `selectNextQueuedBlock`(큐 선두 자동) 대신 `selectQueuedBlocksForTask`(그 과제의 큐 전체)를 조회해, 길이가 1이면 지금과 동일하게 자동 노출(회귀 없음), 2개 이상이면 각 조각을 옵션으로 나열한다. **PredictPage/FocusPage는 손대지 않는다** — 선택은 "큐 재정렬"로 구현하므로(아래 2), 이후 화면들은 여전히 "큐 선두"만 보면 된다.
2. **"선택" = 큐 재정렬(승격), 새 상태 필드 아님.** `block-queue-slice.ts`에 `promoteQueuedBlock(taskId, blockId)`를 추가해 지정한 블록을 그 과제 큐의 맨 앞으로 옮긴다(다른 과제의 큐·상대 순서는 불변). 탭 즉시 이 액션을 호출하고 바로 `/predict`로 이동 — 별도 "확정" 버튼 없음(탭이 곧 확정, CLAUDE §4 마찰 최소화). `PredictPage`의 기존 `selectNextQueuedBlock` 로직이 승격된 블록을 자동으로 다음 것으로 인식하므로 `PredictPage` 자체는 무변경.
3. **옵션 목록 순서 = 쪼갠 순서 그대로, 재배열 UI 없음.** 우선순위 판단을 재도입하지 않기 위해(D-05 근거) 목록은 항상 큐에 쌓인 순서(=쪼개기 화면에서 입력한 순서)로만 보여준다. 정렬·필터·중요도 표시 일절 없음.
4. **영점조절 3버튼은 `retro-context-slice`에 `timeSenseFeedback` 필드로 추가한다, Storage 확장 아님.** `capturedThought`와 동일한 패턴(인메모리, `RetroPage` 언마운트 시 정리) — 이번 위상은 텔레메트리 파이프라인(PH-10)을 새로 만들지 않으므로 영속·집계 없이 "선택했다는 사실"만 그 세션 동안 잠깐 들고 있다가 사라진다.
5. **3버튼은 완료/미완료 양쪽에 공통 렌더하고, 어떤 진행 버튼도 막지 않는다.** D-11 근거("완료 여부와 무관한 별개 축")를 그대로 따른다 — 선택 안 하고 바로 "바로 다음 블록"/"이어서 15분 더"/"오늘은 여기까지"를 눌러도 정상 동작해야 한다(강제 응답 요구 금지, 결정 피로 차단과 침묵 규칙 둘 다와 정합).
6. **PH-06 산출물(딴생각 포착 카드·세션 복구·일시정지) 및 PH-07 온보딩 게이트는 국소 확장 지점만 건드리고 로직은 무변경.** `RetroPage`의 신규 3버튼은 `RecognitionChip` 아래·`BonusCard` 위에 삽입(SPEC §3 트리 순서: 영점조절 → 예측보너스 → 보상/이월과 일치), `CapturedThoughtCard`·클린업 훅은 그대로 재사용·확장만.

## In-Scope

**A. 블록 큐 셀렉터·액션 확장**

- [x] `src/lib/core-loop-selectors.ts` — `selectQueuedBlocksForTask(queuedBlocks, taskId): QueuedBlock[]` 순수 함수(해당 과제의 큐를 순서대로 전부 반환) + 유닛 테스트(0/1/2+ 개, 다른 과제 큐 비간섭)
- [x] `src/store/slices/block-queue-slice.ts` — `promoteQueuedBlock(taskId, blockId): void` 액션(지정 블록을 그 과제 큐 맨 앞으로 이동, 다른 과제 큐·상대 순서 불변) + 유닛 테스트(중간 항목 승격, 이미 맨 앞인 항목 승격 시 무변화, 존재하지 않는 id는 무시)

**B. 대시보드 자기선택 (`DashboardPage.tsx`, `OptionRow` 재사용)**

- [x] `selectQueuedBlocksForTask` 결과 길이 `=== 1` → 기존 동작 그대로(자동 노출 `TaskCta`, 선택 UI 렌더 0 — 회귀 없음)
- [x] 길이 `>= 2` → 안내 문구("어떤 조각부터 해볼까요? 만만한 걸로 골라봐요" 류, 평가어·재촉 없음) + 큐 순서 그대로 `OptionRow` 나열
- [x] 옵션 탭 → `promoteQueuedBlock(taskId, id)` 호출 즉시 `navigate(ROUTES.predict)`(확인 버튼 없음, 탭=확정)
- [x] One Task 어서션 유지: 선택 UI 렌더 중에도 주 과제 카드(`[data-task-card]`) DOM 카운트 `=== 1`(옵션들은 그 카드 내부 하위 요소)
- [x] 유닛 테스트: 큐 1개(선택 UI 없음, 기존 회귀 테스트 그대로 통과) / 큐 2개 이상(옵션 개수 === 큐 길이, 임의 옵션 탭 → `/predict` 이동 + 해당 블록이 `selectNextQueuedBlock`으로 승격됐는지 스토어 어서션)

## DO NOT CHANGE (이 위상 국소 — 전역 목록은 README §0 참조)

- `PredictPage`·`FocusPage`의 내부 로직·라우팅(큐 재정렬만으로 자기선택을 해소하므로 이 두 화면은 무변경)
- `Storage` 인터페이스, `TimerSlice`·`PredictionSlice`·`EnergySlice` 계약
- PH-06이 만든 `judgeSessionReturn`·`useSessionRecovery`·일시정지(길게 누름)·`CapturedThoughtCard` 로직
- PH-07의 `isOnboardingComplete` 라우팅 게이트·`OnboardingPage`
- 디자인 토큰 값(`action`/`evidence.fill`) — `OptionRow` 그대로 재사용(대비 상속, 재측정 불요)

## Positive Non-Goals

- **텔레메트리 로깅·집계 없음**(PH-10 몫) — `timeSenseFeedback`은 세션 중 UI 상태로만 존재, 영속·전송 없음
- **D-15 "재검토 트리거"(순식간 반복 관측 시 블록 길이 재논의) 로직 없음** — 수집 UI만, 정책 반응 없음(post-MVP)
- **선택 목록 정렬·필터·중요도 표시 없음** — 항상 쪼갠 순서 그대로, "만만함"은 사용자 직감에 맡기고 앱이 힌트를 주지 않음(D-05 정신 유지)
- **새 라우트/화면 없음** — 기존 Dashboard/Retro 국소 확장만
- **3버튼 강제 응답 없음** — 스킵 가능, 검사형 흐름 아님
