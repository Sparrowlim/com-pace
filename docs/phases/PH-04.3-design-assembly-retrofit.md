# PH-04.3 — 디자인 조립 규약 소급 적용 (Design Assembly Retrofit)

> **의존:** [PH-04.2](PH-04.2-design-assembly-spec.md)
> **SSOT:** `docs/DESIGN-SYSTEM.md`(PH-04.2 산출물, 4절: 여백·타이포·깊이·모션) · `docs/DESIGN-TOKENS.md §10-6`(기계 소스) · [phases/README.md §0](README.md#0-전역-규칙) — DO NOT CHANGE·Runnable State 정의는 여기 반복 안 함.
> **전역 규칙:** [phases/README.md §0](README.md#0-전역-규칙) 참조.

## Goal

PH-04.2가 만든 조립 규약(`DESIGN-SYSTEM.md` 4절)을 나머지 UI 프리미티브 5종(`Button`·`Chip`·`EnergyBar`/`EnergyCell`·`OptionRow`·`BottomSheet`)과 9개 페이지에 소급 대조하고, stylelint로 간격/색 하드코딩을 기계 차단하는 상태. `TaskCard`는 PH-04.2에서 이미 대조 완료(편차 없음) — 이 위상에서는 재확인만.

## SSOT 발췌 (착수 직전 필수 — 원문 미개봉 자족성)

**`DESIGN-SYSTEM.md`가 PH-04.2 시점에 이미 기록해둔 관찰(이 위상이 처리할 명시적 대상):**

> §1 여백 리듬: "`Button`의 가로 패딩(`space.6`)은... `Chip`의 가로 패딩(`space.5`)도... '카드-내부' 계층 토큰(`space.3`~`space.4`)으로 정렬하는 것이 원칙에 맞다."

> §3 깊이 규칙: "`Button.primary`·`Chip.selected`·`OptionRow`는 현재 elevation을 전혀 쓰지 않는다... 실제 적용은 6종 소급 감사(`PH-04.3`)에서 처리한다."

**컴포넌트 매핑표(§3, 재인용 — 이 위상의 elevation 목표값):**

| 컴포넌트                 | 서열 | 토큰                       |
| ------------------------ | ---- | -------------------------- |
| `EnergyBar`/`EnergyCell` | 0    | — (그림자 없음, 의도)      |
| `OptionRow`              | 1    | `elevation.inner`          |
| `Chip`(`selected`만)     | 2    | `elevation.soft`           |
| `TaskCard`               | 3    | `elevation.card`(기 적용)  |
| `Button`(`primary`만)    | 5    | `elevation.cta`            |
| `BottomSheet`            | 6    | `elevation.sheet`(기 적용) |

**여백 3계층 재인용(§1):** 카드-내부(요소 간) = `space.1`~~`space.4`(4/6/8/12px) · 카드-간(형제 항목 사이, "리스트/스택에서 반복되는 형제 카드·옵션 행 사이 간격") = `space.5`~~`space.6`(16/20px) · 섹션-간(카드 자체 외곽 패딩) = `space.7`~`space.8`(24/30px).

## 소급 감사 결과 (이 세션에서 5개 컴포넌트 CSS + 9개 페이지 CSS/TSX 전수 직접 대조 완료)

> `phases/README.md §0`의 "읽기 전용 서브에이전트 병렬 감사" 절차 대신, 대상이 CSS 파일 14개(컴포넌트 5·페이지 9)로 크지 않아 이 세션에서 직접 전수 대조했다(서브에이전트 위임보다 판단 정확도가 높음 — 여백 계층 배정은 컴포넌트 역할에 대한 맥락적 판단이 필요해 기계적 grep으로 대체 불가).

## DO NOT CHANGE (이 위상 국소)

- `tokens/design-tokens.json`·`DESIGN-TOKENS.md`의 모든 값 — 배치(CSS 선언)만 조정, 토큰 재정의 금지
- `action`/`evidence.fill` 및 모드 오버레이 — 어떤 CSS 변경도 이 토큰의 모드 재정의를 유발하면 안 됨
- 컴포넌트 TSX(props·마크업·이벤트 핸들러) — 이 위상은 `.module.css`만 건드린다
- `OptionRow`의 내부 패딩(`space-4 space-5`) — `DESIGN-SYSTEM.md`가 이 컴포넌트를 spacing 관찰에서 명시적으로 언급하지 않았으므로, 이 위상에서 새로 판단해 값을 바꾸지 않는다(SSOT 범위 밖 재해석 금지 — 필요하면 별도 관찰로 기록 후 차기 위상에서 다룸)
- 페이지의 `.actions`(버튼 스택) 간격 — "형제 카드·옵션 행"의 명시적 정의에 해당하지 않아 재해석해 변경하지 않음

A(컴포넌트 5종)·B(페이지 4개)·C(stylelint)·D(검증) 전 항목 구현: `Button`/`Chip` 내부 패딩을 카드-내부 계층(`space-4`)으로 정렬 + `Button.primary`(`elevation.cta`)·`Chip.selected`(`elevation.soft`)·`OptionRow`(`elevation.inner`) box-shadow 추가, `DashboardPage`/`PredictPage`/`RetroPage`의 `OptionRow` 형제 리스트와 `SplitPage`의 `draftRow` 리스트 gap을 카드-간 계층(`space-5`)으로 정렬. `stylelint`(devDependency) + `stylelint.config.js`(`declaration-property-value-disallowed-list`로 margin/padding/gap의 raw px, color 계열의 raw hex/rgba 차단, `tokens.generated.css` 제외) 신설, `lint:css` 스크립트 + `.claude/hooks/lint-on-write.cjs`에 `.css` 분기 추가. 검증 중 `npm install`이 플로팅 `eslint` semver 범위(`^9.18.0`)를 9.39.4로 끌어올리며 `e2e/task-card.spec.ts`의 Playwright 공식 fixture-skip 관용구(`({}, testInfo)`)에서 `no-empty-pattern` 신규 에러가 발생 — PH-04.3 범위 밖이지만 Runnable State를 막아 `eslint-disable-next-line` 1줄로 해소(회귀 아님, 사전 상태로 복귀). `Button` 패딩/elevation 변경으로 커밋된 `e2e/task-card.spec.ts-snapshots/*.png`(온보딩 CTA 포함) 픽셀 diff 발생 — diff 이미지 육안 확인 결과 버튼 자체의 의도된 시각 변경뿐 가로 스크롤·레이아웃 밀림 없음 확인 후 `--update-snapshots`로 베이스라인 재생성. 비커밋 임시 Playwright 스크립트로 온보딩→쪼개기(Chip)→대시보드(FragmentChoice OptionRow)→예측(OptionRow) 실제 클릭 왕복 확인(320px, 가로 스크롤 0, 완료 후 스크립트 삭제). 전체 306개 vitest 통과, typecheck/lint(경고 6건, 에러 0)/lint:css/build 전부 exit 0. `OptionRow` 자체 내부 패딩과 페이지 버튼 스택 간격은 SSOT 미언급이라 의도적으로 무변경 유지.
- **v0.2** — 착수 직전 상세화(30~50개 체크리스트, 계획만 — 구현 아직 없음). `DESIGN-SYSTEM.md`/`DESIGN-TOKENS.md`가 PH-04.2 시점에 이미 기록해둔 관찰(Button/Chip 여백 계층, Button.primary/Chip.selected/OptionRow elevation 누락)을 SSOT로 그대로 승계 + 이 세션에서 5개 컴포넌트·9개 페이지 CSS/TSX를 직접 전수 대조해 추가 편차 4건 발견(DashboardPage/PredictPage/RetroPage의 형제 OptionRow 리스트, SplitPage의 draftRow 리스트 — 전부 "카드-간" 계층 오배정). 서브에이전트 병렬 감사 대신 이 세션에서 직접 대조한 이유: 대상 CSS가 14개 파일로 크지 않고, 여백 계층 배정에는 컴포넌트 역할에 대한 맥락적 판단이 필요해 정확도상 직접 대조가 낫다고 판단. OptionRow 자체 내부 패딩과 페이지의 버튼 스택 간격은 SSOT가 명시적으로 언급하지 않은 항목이라 이 위상에서 재해석해 변경하지 않기로 결정(DO NOT CHANGE에 근거 기록).
- **v0.1** — 최초 작성(헤더만). PH-04.2에서 분리 — 소급 감사 범위가 착수 전엔 미확정이라 상세 체크리스트를 지금 쓰면 추측성 과설계(`phases/README.md §0` 원칙 위반)이므로 헤더만 등록.
