## Goal

SCREEN-FLOW.md/SPEC.md와 실제 구현 간 2z·3-A·6-A 드리프트를 해소하고, 사용자가 실제 앱 구동 중 보고한 "회고 화면(6 계열) 자체가 안 보인다"는 리포트의 근본원인을 확인해 처리했다.

## 배경 — 이번 세션 진단 요약

- **원 이슈:** SCREEN-FLOW.md의 화면 6·7과 연결된 **6-A(휴식)** 화면이 미구현으로 확인됨 — `routes/paths.ts`에 대응 라우트 없음, `RetroPage.tsx`의 완료 경로(`RetroActions`, completed=true)가 "바로 다음 블록" 단일 버튼만 제공하고 "잠시 쉬기" 선택지가 없음.
- **원인 진단(사용자 가설 확인됨):** 두 가지가 복합 작용했다.
  1. `PH-05-core-loop.md` v0.2 상세화 시점(2026-07-06/07)은 `_TEMPLATE.md`의 "SSOT 발췌" 필수화(v0.4, 2026-07-08 소급 감사로 신설)보다 **이전**이라, "2z·6-A·3-A는 SSOT 화면 목록에 없다"는 판단을 SCREEN-FLOW.md 원문 재대조 없이 내렸다(실제로는 SCREEN-FLOW.md §1에 두 화면 모두 행이 있었다).
  2. 그 판단이 PH-05 파일에만 "착수 전 설계 결정 5"로 기록되고 **SCREEN-FLOW.md 원문에는 되먹임되지 않아**, 이후 `PH-04.4`·`DESIGN-SYSTEM.md`가 그대로 인용하며 굳어졌다.
- **재발 패턴 확인:** 같은 유형("SSOT 재확인 없이 스코프컷을 자체 판단으로 내리고 SSOT에 되먹임 안 함")의 실패가 이 프로젝트에서 이미 3회 확인됨 — 자기선택 누락(D-05, PH-05.1로 수정), 영점조절 누락(D-11, PH-05.1로 수정), 이번 2z/6-A/3-A.

## 합의된 방침

1. **Phase A(SSOT 드리프트 수습)는 SCREEN-FLOW.md 원문(§1 인벤토리·§2 Mermaid·§3 상태전이표)을 직접 근거로 삼는다** — PH-05가 자체 기록한 "SSOT에 없다"는 주장을 신뢰하지 않고 원문을 다시 연다(사용자 지시: "화면흐름 표 기준으로 작업").
2. **재확인 결과 — 2z(빈 대시보드)·3-A(새 과제 추가)는 이미 해소된 상태였다.** `PH-04.4-component-catalog.md`(86행)·`DESIGN-SYSTEM.md`(138행)가 이미 "대시보드 `AddTaskPrompt` 하나가 1-A/2z/3-A 3개 화면 상태를 겸함 — 별도 컴포넌트 불필요"로 명시적 판정을 내려뒀다. `DashboardPage.tsx`의 `!task` 분기(`AddTaskPrompt`)가 이 3개 화면 ID의 기능적 요구를 전부 충족한다(과제 소진 상태 노출 + 새 과제 입력을 별도 화면 전환 없이 인라인으로 처리 — CLAUDE §4 마찰 최소화에도 부합). **코드 변경 불필요.** SCREEN-FLOW.md §1 배지·§7 "다음 단계" 목록·SPEC.md §4의 "별도 와이어 필요" 문구를 이 판정에 맞춰 정정 완료(v0.4).
3. **6-A(휴식)는 실재하는 갭이다.** PH-05 착수 전 설계 결정 5에서 의도적으로 생략됐고, 이후 어떤 위상도 이를 후속 과제로 배정받지 않은 채 방치됐다.

## ✅ 재현·근본원인·수정 (완료)

> **15분 블록 종료 직후 회고 화면(6/6′/7/7′) 자체가 보이지 않고, 곧장 대시보드로 넘어간다.**

- **재현:** dev 서버(`npm run dev -- --port 5183`) 기동 + Playwright로 온보딩→쪼개기→예측→집중(DevSkip)→종료 실제 왕복. `finish(true)` 클릭 직후 URL이 `/retro`를 전혀 거치지 않고 곧장 `/`(대시보드)로 확인됨 — 리포트 그대로 재현.
- **근본원인:** React 18 `<StrictMode>`(`src/main.tsx`)는 개발 모드에서 마운트 직후 effect를 **클린업→재실행**으로 한 번 더 시뮬레이션한다(실제 언마운트 아님). `RetroPage.tsx`의 `useClearRetroContextOnUnmount`는 클린업에서 `setLastResolvedBlock(null)`을 호출하는데, 이 클린업이 그 시뮬레이션 중 곧바로 실행되면서 스토어의 `lastResolvedBlock`이 마운트 직후 null로 되돌아갔다. `RetroPage`의 가드(`if (!lastResolvedBlock) return <Navigate to={ROUTES.dashboard} />`)는 이 순간적 null에 반응해 대시보드로 리다이렉트했다 — 사용자는 회고 화면을 볼 틈이 없었다.
  - **검증:** `main.tsx`에서 `<StrictMode>`를 임시로 제거하고 동일 재현 스크립트를 재실행한 결과 `/retro`에 정상 체류함을 확인(가설 확정). 이후 `main.tsx`는 원상복구.
- **수정(`RetroPage.tsx`):** 라이브 구독(`useAppStore((s) => s.lastResolvedBlock)`) 대신 **마운트 시점 1회 스냅샷**(`useState(() => useAppStore.getState().lastResolvedBlock)`)을 가드·렌더 근거로 사용하도록 교체. StrictMode의 시뮬레이션이 스토어 값을 순간적으로 null로 되돌려도 로컬 스냅샷은 영향받지 않는다. 언마운트 시 스토어를 정리하는 기존 로직(`useClearRetroContextOnUnmount`)은 그대로 유지 — 실제 이탈(뒤로가기 후 재방문 시 스테일 데이터 방지)에는 여전히 필요하다.
  - StrictMode 재확인 재현(수정 후, `<StrictMode>` 유지 상태)으로 `/retro` 체류 + 정상 렌더(헤드라인·영점조절 3버튼·보너스카드·"바로 다음 블록") 확인.
  - **스코프 경계:** 같은 클린업이 `capturedThought`도 null로 되돌리지만, 이 값은 개발 모드에서만(StrictMode는 프로덕션 빌드에 없음) 순간적으로 사라지는 조용한 이슈이고 가드처럼 화면 자체를 날리지 않는다 — 이번 리포트(회고 화면 미노출)의 범위 밖으로 남겨둔다(YAGNI, 별도 리포트 없음).

  ```
  DONE_HIT -->|"잠시 쉬기"| REST["6-A 휴식"]
  DONE_MISS -->|"잠시 쉬기"| REST
  DONE_HIT -->|"바로 다음 블록"| NEXT{{"다음 블록 있나?"}}
  DONE_MISS -->|"바로 다음 블록"| NEXT
  ```

### 착수 전 설계 결정

- **D1. 5분 강제 타이머 없음.** §1 인벤토리의 "5분 휴식"은 원 와이어프레임 캡션의 뉘앙스 표현일 뿐, SPEC.md·DECISIONS.md 어디에도 기계적 카운트다운 명세가 없다. 오히려 [`D-11`](../DECISIONS.md#d-11)은 "휴식 권장 단독"을 대안으로 검토 후 기각(시간 영점조절 3버튼 채택)한 이력이 있다. 새 타이머는 CLAUDE §2 "15분 단일 고정"(유일한 고정 시간 구조) 밖의 신규 시간 구조라 스코프 밖(YAGNI) — 자유 이탈 가능한 2버튼으로 구현한다.
- **D2. `RestPage`는 진입 가드 없음.** `PredictPage`/`DischargeEntryPage`와 달리 activeBlock 존재 여부를 확인할 필요가 없다 — 도달 시점(방금 블록 종료 직후)엔 activeBlock이 이미 없다. "다음 블록" 버튼은 `RetroPage.handleNext`와 동일한 `resolveNextRoute` 로직을 재사용해 큐가 비어 있으면 자연히 대시보드로 향하므로, 직접 URL 진입에도 크래시 없이 안전하다.
- **D3. 카피는 자체 창작.** CLAUDE §8(원문 문구·명칭 복제 금지)에 따라 와이어프레임 캡션을 그대로 쓰지 않고 새로 쓴다. 톤은 CLAUDE §4(따뜻한 큰 어른, 과정 칭찬, 잔소리·검사 금지).
- **D4. 완료(6/6′) 경로에만 추가, 미완료(7/7′)는 무변경.** Mermaid 근거는 위 SSOT 발췌 참조.
- **D5. 컴포넌트 카탈로그 게이트 — 신규 등재 불요.** `RestPage`는 `Button` + 안내 문구 조합으로 `DischargeEntryPage`(카탈로그 등재 완료)와 동일 조립 패턴을 재사용한다. 신규 UI 패턴이 아니므로 `DESIGN-SYSTEM.md §5` 추가 등재 없이 진행한다.
