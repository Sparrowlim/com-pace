# PH-04.2 — 디자인 조립 규약 (Design Assembly Spec)

> **의존:** [PH-04](PH-04-tokens-ui.md)(완료)
> **SSOT:** [`DESIGN-TOKENS.md §10-6`](../DESIGN-TOKENS.md#10-미결--튜닝-여지-브리프-7-유예-잔여)(컴포넌트 토큰 "다음 단계" 예고) · [`DESIGN-BRIEF.md`](../DESIGN-BRIEF.md) · [`CLAUDE.md §8`](../../CLAUDE.md)(법적 원칙 — 원리 차용/원문 복제 금지 패턴을 시각 시스템에도 재사용) · [`DECISIONS.md`](../DECISIONS.md) 부록A(법적·IP 표 형식 재사용)
> **참고 스킬:** `frontend-design`(distinctive design 판단 시), `a11y-architect`(터치 타깃/대비 재확인 시)
> **전역 규칙:** [phases/README.md §0](README.md#0-전역-규칙) DO NOT CHANGE·Runnable State 정의는 여기 반복 안 함.

## Goal

`docs/DESIGN-SYSTEM.md`(HIG 원리를 컴페이스 토큰 위에 매핑한 조립 규약)와 `DECISIONS.md D-27`(차용/거부 경계 ADR)이 존재하고, `TaskCard` 1개 컴포넌트가 그 규약과 대조·검증된 상태. **SCREEN-FLOW/SPEC 행동 변경 없음** — 순수 조립·문서 위상.

## 배경 (SSOT 발췌 — 원문 미개봉 자족성)

> `DESIGN-TOKENS.md §10-6` 항목 6: "컴포넌트 토큰(다음 단계) — `Button`·`Chip`·`TaskCard`·`EnergyBar`·`OptionRow`·`BottomSheet`를 이 semantic 위에 얹으면 SCREEN-FLOW 재구성이 '조립'만 남는다." 이 위상이 그 "다음 단계"를 시작한다.

> `CLAUDE.md §8`: "기획은 논문에서(What), 구현은 백지에서(How). 원리는 차용, 원문 문구·명칭·도식은 복제 금지." 이 위상이 Apple HIG에 대해 동일 원칙을 적용한다 — **시각 정체성(SF Pro·시스템 블루·iOS 크롬)은 이식하지 않고, 위계·리듬·깊이·모션 절제의 원리만 차용한다.**

## 원리 커버리지 표 (SPEC 커버리지 게이트의 이 위상용 변형)

> 이 위상은 `SCREEN-FLOW.md` 상태전이를 다루지 않는다(사용자 행동 변화 없음). 대신 사전 정합성 점검에서 확정한 "HIG 차용/거부 판정"의 각 항목을 In-Scope/설계 결정/Non-Goal 중 정확히 하나에 배정한다 — 세 곳 중 어디에도 없는 항목이 남으면 상세화 미완료.

## In-Scope

**A. 문서**

- [x] `docs/DECISIONS.md`: `D-27` 신설(차용 원리 vs 거부 항목, K=Android(`D-26`) 정합 근거 포함), 인덱스 표에 행 추가, 부록A "법적·IP" 표에 `Apple HIG | 위계·여백·깊이·모션의 원리 | SF Symbols·시스템 컬러·명칭·도식 원문` 행 추가
- [x] `docs/README.md`: 문서 지도 표에 `DESIGN-SYSTEM.md` 행 추가(`DESIGN-TOKENS.md`↔`DESIGN-BRIEF.md` 사이, "화면/컴포넌트 구현 직전" 로드 시점), 위계 다이어그램·정본 소유표에 "화면 조립 규약" 사실 추가
- [x] `docs/DESIGN-TOKENS.md §10-6` 항목 6("컴포넌트 토큰(다음 단계)")을 완료 처리, `docs/DESIGN-SYSTEM.md` 링크로 교체
- [x] `docs/DESIGN-SYSTEM.md` 신규 작성(서두에 위계 문구 필수: "본 문서는 `CLAUDE.md §2` 불변 규칙·`DB-02` 하드 규칙을 깨지 못한다"):
  - [x] **여백 리듬** — `space.*` 8종을 카드-내부/카드-간/섹션-간 3계층 용도로 고정(현재 컴포넌트마다 임의 선택 중인 상태 해소)
  - [x] **타이포 위계** — 카드/화면 1개 안에서 동시 사용 가능한 `font.size` 조합 ≤3단계(주·보조·라벨) 규칙, 세리프는 제목 1곳 한정(`DB-03` 재확인)
  - [x] **깊이 규칙** — `elevation.*` 6종을 "떠있는 정도" 순서로 서열화 + 컴포넌트별 매핑표(기존 `TaskCard=card`·`BottomSheet=sheet` 포함 6종 전수)
  - [x] **모션/피드백 일관성 표** — 상태 변화(선택·완료·에너지 점등 등)마다 `duration`/`easing` 조합을 고정한 컴포넌트 횡단 표

**B. 파일럿 검증 (규약이 실제로 조립 가능한 수준인지 소규모로 먼저 검증 — 6종 동시 착수 전 리스크 절감)**

- [x] `TaskCard.module.css`를 새 규약과 대조, 편차 있으면 배치만 수정(토큰 값 변경 없음)
- [x] 기존 `TaskCard.test.tsx` 무회귀 통과
- [x] Playwright 스크린샷(320/390/768px) 재캡처 — 새 시각 기준선을 세우는 작업이라 이 항목만 예외적으로 육안 1회 확인 허용(그 외 전부 기계 검증)

## DO NOT CHANGE (국소)

- `tokens/design-tokens.json`·`DESIGN-TOKENS.md`의 모든 값(§9 JSON) — 이 위상은 배치 규칙만 추가, 값 재정의 금지
- `action`/`evidence.fill` 및 그 semantic 매핑 — 어떤 문서·컴포넌트 변경도 이 두 토큰의 모드 재정의를 유발하면 안 됨
- PH-00~09 산출물 중 `TaskCard` 외 컴포넌트/페이지(→ PH-04.3 범위, 이 위상에서 손대지 않음)

## Positive Non-Goals

- `Button`·`Chip`·`EnergyBar`·`OptionRow`·`BottomSheet` 및 9개 페이지 소급 감사(→ [PH-04.3](PH-04.3-design-assembly-retrofit.md))
- stylelint 등 기계 강제 도입(→ [PH-04.3](PH-04.3-design-assembly-retrofit.md))
- 토큰 값 자체의 재조정(`evidence.fill` 비텍스트 대비 미해결 항목 포함, `DESIGN-TOKENS §10-4` 그대로 유지)
