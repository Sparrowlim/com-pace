# 화면 구성 (Composition) — `CMP`

> 상위: [`../DESIGN-SYSTEM.md`](../DESIGN-SYSTEM.md) 허브 · 간격 크기: [`spacing.md`](spacing.md) · 화면 원형별 적용: [`decision-guide.md`](decision-guide.md)
> 이 문서는 **"요소를 화면 세로 어디에 두는가"**를 소유한다. [`spacing.md`](spacing.md)가 "얼마나 벌리는가(간격 크기)"라면, 이 문서는 "어디에 두는가(배치·무게중심)"다.
> **위계:** 아래 어떤 배치 규칙도 CLAUDE §2(담백)·§3(도파민 anti-goal)·§6-6(볼거리 금지)을 깨지 못한다. 특히 **CMP-6**을 먼저 읽는다.

## 왜 이 문서가 생겼나

화면이 "비어 보이거나 뭉쳐 보이는" 편차 — 콘텐츠가 상단에만 쌓이고 가운데가 목적 없이 비며 하단에 요소 하나가 외롭게 떠 있는 **바벨(barbell) 레이아웃** — 은 배치 규정의 공백에서 나온다. 세로 배치를 "버튼 y좌표 고정" 한 가지 관심사로만 다루면(구 §1-1), 그 부작용으로 정의되지 않은 공허가 생산된다. 이 문서는 앵커 규칙을 **폐기하지 않고 위로 한 겹 올려**, 공허를 담백한 여백으로 프레임하거나 좁히는 규정을 더한다.

## CMP-1. 세로 3존 (모든 화면 공통 골격)

모든 화면은 위→아래 세 존으로 조립한다.

| 존         | 내용                                         | 배치                                         |
| ---------- | -------------------------------------------- | -------------------------------------------- |
| **헤더존** | 북극성 배지·설정 진입·방전 종료 배너 등 크롬 | 화면 최상단 고정                             |
| **초점존** | 그 화면의 대표 콘텐츠(카드·질문·입력·타이머) | 헤더 바로 밑 밀착 ❌ — CMP-3 초점 밴드       |
| **앵커존** | 주 행동 버튼군 + 그날의 증거(`EnergyBar`)    | 뷰포트 하단 고정(`margin-top: auto`) — CMP-2 |

## CMP-2. 앵커 불변 (ADHD 공간 일관성)

주 행동 버튼(그 화면의 흐름을 전진시키는 CTA — 시작하기·다음·완료·돌아가기 등)은 **화면 종류와 무관하게 항상 앵커존(뷰포트 하단 고정)**에 둔다. 콘텐츠 길이와 무관하게 버튼 y좌표가 흔들리지 않아야 한다 — 실행 기능 저하 상태(페르소나 K, CLAUDE §1·§4)에서 "버튼이 이번엔 어디 있지"를 매번 다시 찾는 것 자체가 마찰이기 때문이다.

```css
.page {
  display: flex;
  flex-direction: column;
  min-height: 100svh;
  padding: var(--space-6);
}
.focal {
  margin-top: auto; /* 초점존 — 남는 세로 공간 안 수직 중앙, CMP-3 */
  margin-bottom: auto;
}
.actions {
  /* 앵커존 — margin-top:auto 불필요. .focal이 위/아래 auto 마진으로 자유 공간을
     전부 흡수하므로, 마지막 자식인 이 요소는 그 결과로 항상 뷰포트 최하단에 붙는다. */
}
```

- **"콘텐츠+버튼 통짜 중앙정렬"은 여전히 금지(v0.7에서 폐기).** 버튼을 초점 콘텐츠와 한 덩어리로 묶어 화면 중앙에 두면, 콘텐츠 길이가 화면마다 달라질 때 버튼의 실제 y좌표도 함께 흔들려 이 문서의 핵심 원칙과 충돌한다 — 이건 지금도 하지 않는다.
- **카드만 남는 공간에서 수직 중앙에 두는 것은 별개(2026-07-17, 허용).** 위 스니펫처럼 초점존과 앵커존이 이미 별도 flex 자식으로 분리돼 있으면, 초점존에만 `margin-top`/`margin-bottom` 양쪽 `auto`를 줘도 앵커존의 y좌표는 흔들리지 않는다 — 자유 공간이 초점존 쪽에서만 소진되고 앵커존은 그 결과로 여전히 최하단에 고정되기 때문이다(버튼 위치 불변이라는 실제 목적은 유지, 카드가 헤더 바로 아래 고정되며 생기던 "바벨 공허"만 해소). 사용자 피드백(카드-버튼 사이 공백 과다)으로 CMP-3의 "상단 고정" 배치를 이 방식으로 교체했다.
- **단일 규칙(카드 유무 무관).** 이전엔 "카드 없는 화면(하단 고정)"과 "`TaskCard`로 감싸는 화면(카드 내부 CTA, 상단 고정)"을 별도 그룹으로 두고 그 절대 위치 편차를 의도적으로 유지했으나, 2026-07-17 재구조화로 이 예외를 없앴다. 이제 모든 화면이 동일한 3존 골격을 따른다:
  - **`TaskCard`는 카드 콘텐츠(제목·설명·다음 조각 라벨·선택지)만 담당**하고, 흐름을 전진시키는 CTA를 자신의 children으로 그리지 않는다. 카드는 헤더 바로 아래 고정이 아니라 다른 초점 콘텐츠와 동일하게 CMP-3 초점 밴드(남는 공간 안 수직 중앙)에 위치한다.
  - **주 CTA는 카드 밖, 앵커존에서만** 렌더한다. 적용: `OnboardingPage`·`SplitPage`·`DashboardPage`(카드 상태)·`DischargeDashboardPage` — 기존에 이미 이 패턴이던 `NorthStarPage`·`PredictPage`·`SettingsPage`·`RestPage`·`DischargeEntryPage`·`RetroPage`와 동일 규칙으로 통합. 단 이 6개 화면은 카드형이 아니라 콘텐츠가 상단부터 자연스럽게 흐르는 별개 패턴(입력형·선택형·목록형, `decision-guide.md` DG-1)이라 이번 수직 중앙 개정 대상이 아니다 — 카드형(`TaskCard` 보유) 4화면만 해당.
  - **예외:** 탭 자체가 확정 행위인 자기선택 UI(대시보드 `FragmentChoice`, PH-05.1)는 "흐름을 전진시키는 CTA"가 애초에 없다 — 억지로 앵커존에 버튼을 만들지 않는다. 이건 위치 편차가 아니라 애초에 CTA가 없는 별개 패턴이다.
- **하드 함정 — `EnergyBar`는 액션과 같은 앵커 컨테이너 안에 있어야 한다.** CMP-1은 앵커존을 "주 행동 버튼군 + 그날의 증거(`EnergyBar`)"로 정의하지만, `margin-top: auto`를 액션 블록(`.actions`) 자신에게만 주면 이 문장이 텍스트로만 참이 되는 함정이 있다 — `EnergyBar`가 `.actions`보다 앞서 렌더되면 그 자유 공간 흡수는 `.actions`만의 몫이라 `EnergyBar`는 위쪽 초점 콘텐츠 흐름에 남고, 실제로는 앵커존이 아니라 화면 상단에 붙은 채 버튼과만 크게 벌어진다(`RetroPage` v0.14에서 실측 스크린샷으로 발견). 해법은 `EnergyBar`+액션을 감싸는 별도 그룹(`DashboardPage`의 `.bottomGroup`과 동일 패턴)을 만들고, `margin-top: auto`를 그 그룹에 주는 것 — 그룹 안에서 `EnergyBar`는 액션 바로 위, 작은 `gap`만 유지한 채 함께 하단에 고정된다.

## CMP-3. 초점 밴드 (focal band)

초점 콘텐츠(대표 카드·질문·입력·타이머)를 **헤더 바로 밑에 밀착시키지 않는다.** 카드형 화면(`TaskCard` 보유 — `OnboardingPage`·`SplitPage`·`DashboardPage`(카드 상태)·`DischargeDashboardPage`)은 초점존과 앵커존 사이 남는 세로 공간 안에서 초점 콘텐츠를 **수직 중앙**(`margin-top`/`margin-bottom` 양쪽 `auto`, CMP-2 스니펫 참조)에 둔다 — 콘텐츠가 "위에 처박힌" 느낌도, 카드-버튼 사이가 텅 빈 "바벨 공허"도 아닌 **"무대 가운데 놓인" 느낌**이 되게 한다.

- 카드 없는 화면(입력형·선택형·목록형)은 이 수직 중앙 규칙 대상이 아니다 — 콘텐츠가 상단부터 흐르고 앵커존만 `margin-top: auto`로 하단 고정된다(CMP-2 v0.7 이후 패턴 그대로).
- 이 배치는 앵커(CMP-2)와 충돌하지 않는다 — 초점존 *안에서만*의 배치이고 앵커존은 여전히 자유 공간과 무관하게 최하단에 고정된다.
- 콘텐츠가 길어지는 화면(예: `SplitPage`의 조각 초안 목록)에서는 카드 자체가 커지며 남는 공간을 자연히 흡수하므로, 짧은 콘텐츠일 때만 위/아래 여백이 뚜렷하게 보인다 — 의도된 동작이다(CMP-4).

## CMP-4. 공백은 프레임되어야 한다 (void must be framed)

초점존과 앵커존 사이의 빈 공간은 **결함이 아니라 담백함의 여백일 수 있다.** 단 아래 두 조건을 모두 만족할 때만 여백으로 읽히고, 아니면 "고장"으로 읽힌다.

1. **초점 콘텐츠가 충분한 presence를 가진다** — 대표 카드는 옹색한 최소 크기가 아니라 생성적 내부 여백(외곽 패딩 `space.7`, 제목-본문-액션 사이 카드-내부 계층)으로 "의도된 무대"의 무게를 갖는다. 짧은 콘텐츠("123" 한 줄)여도 카드 자체는 담백하되 옹색하지 않게.
2. **앵커존이 비어 보이지 않는다** — CMP-5.

"작은 카드 + 보이지 않는 앵커"의 조합은 여백이 아니라 바벨 공허다.

## CMP-5. 앵커존 비가시 방지

앵커존의 요소가 전부 조건부라 사라지면(예: `EnergyBar`가 `filledCount=0`이라 캡션 없음 + `DischargeLink` 부재) 앵커가 텅 빈 것처럼 보인다.

- 앵커존에는 **항상 최소 하나의 가시 요소를 보장**하거나, 없으면 초점존이 그 세로 공간을 자연스럽게 흡수하도록 재구성한다.
- 해법은 **재배치(⭕)**이지 장식 추가(❌)가 아니다. 특히 `EnergyBar`의 빈 상태를 미리 그린 빈 칸으로 메우는 것은 영구 금지(`../DESIGN-TOKENS.md §6` anti-token `color.energy.empty` · SPEC §8 "0에서 자람").

## CMP-6. 채움 ≠ 장식 (하드 경계)

위 어떤 규칙도 **볼거리·틴트·일러스트·플레이스홀더 카드·마스코트로 공백을 메우는 것**을 정당화하지 않는다.

- CLAUDE §3(도파민 슬롯머신 anti-goal) · §6-6(집중 화면 볼거리 금지) · `../DESIGN-TOKENS.md §6`(anti-token) 위반.
- 공백은 오직 **비례(proportion) · presence · 재배치**로만 다룬다.
- 판정 질문: "이 요소가 없어도 화면이 자기 일을 하는가?" 그렇다면 그것은 장식이므로 넣지 않는다.

## CMP-7. 콘텐츠 폭 상한 (와이드 뷰포트)

설계 기준선은 320~430px 폰(페르소나 K)이지만, 앱은 768px+ 태블릿/데스크톱에서도 열린다. `.page`가 뷰포트 전체 폭을 그대로 쓰면 입력창·버튼이 풀블리드로 늘어져 "템플릿처럼 보이는" 문제([`../web/design-quality.md`](../../../.claude/rules/ecc/web/design-quality.md) 안티패턴)와 질문↔선택지 사이 세로 공백 확대가 동시에 발생한다(2026-07-18 UX 검토 스크린샷으로 발견).

- 흐름 화면(`.page`)은 `width: 100%; max-width: var(--content-max-width); margin-inline: auto`를 쓴다(`--content-max-width: 30rem`, `src/index.css`에 정의).
- 320~430px 폰에서는 `max-width`가 뷰포트보다 커 **무영향** — 설계 기준선 렌더는 바뀌지 않는다.
- **예외: `FocusPage`.** 집중 화면은 다크 풀블리드가 의도(§6 볼거리 금지·타이머 몰입)라 이 상한을 적용하지 않는다.
- 이 값은 `space.*` 리듬 토큰이 아니라 배치 상수라 [`spacing.md`](spacing.md) 대신 이 문서가 소유한다(SP-2 "배치는 composition.md 소유" 원칙).
